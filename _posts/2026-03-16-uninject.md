# UNInject – High-Performance Unity Dependency Injection SDK
## Current Version : 1.0.3 

![](https://img.shields.io/badge/unity-2021.3%2B-black)
![](https://img.shields.io/badge/license-MIT-blue)

> **UNInject**는 Unity를 위해 설계된 고성능 의존성 주입 프레임워크임.  
> **Zero-reflection runtime**, **editor 기반 dependency baking**,  
그리고 **엄격한 multi-tier lifecycle 제어**를 목표로 설계되었음.

UNInject는 **editor-time dependency baking**, **expression-tree caching**,  
그리고 **hierarchical scoping**을 결합하여 전통적인 DI 구현에서 흔히 발생하는 **reflection 병목**이나  
**runtime lookup 오버헤드 없이** 매우 효율적인 runtime dependency injection을 가능하게 함.

기존 DI 시스템들이 `FindObjectsOfType` 호출이나 무거운 reflection 초기화에 의존하는 것과 달리
UNInject는 **성능**,  
**결정론적 dependency resolution**, **개발자 사용성**을 중심으로 설계된 **완전한 injection architecture**를 제공함.

# Core Features

### Editor-Backed Bake Architecture

UNInject는 **editor-time dependency scanning 시스템**을 도입했음.

이를 통해 개발자는 **sibling component**나
**child hierarchy dependency** 같은 로컬 의존성을
단순한 inspector 버튼 하나로 매칭할 수 있음.

이 과정에서 **runtime lookup 비용은 전혀 발생하지 않음**.

`ObjectInstaller`는 `[Inject]` attribute를 **개발 단계에서 완전히 resolve (get)**하도록 설계되어 있음.

---

### Zero Reflection Overhead at Runtime

dependency injection 수행 시 **runtime에서 reflection을 전혀 사용하지 않음**.

느린 `FieldInfo.SetValue` 방식을 사용하는 대신
UNInject는 `TypeDataCache`를 이용하여  
Expression Tree를 **native `Action<object, object>` delegate로 컴파일함**.

이 방식으로 인해 object injection 과정에서
**initialization spike나 CPU bottleneck이 발생하지 않도록 설계했음**.

---

### Three-Tier Deterministic Scoping

전통적인 DI 시스템들은 lifecycle 관리가 모호해지는 문제가 자주 발생함.

UNInject는 이를 해결하기 위해
**세 가지 명확한 scope 계층을 강제함**.

* **Global Scope (`MasterInstaller`)**
  → `DontDestroyOnLoad`를 통해 scene 전환 이후에도 유지됨

* **Scene Scope (`SceneInstaller`)**
  → 현재 활성 scene에만 존재하며 scene 전환 시 파괴됨

* **Local Scope (`ObjectInstaller`)**
  → 특정 GameObject root와 그 children 범위로 제한됨

모호한 binding 없음.
메모리 누수 없음.
runtime container 탐색 오버헤드가 없음.

---

### PlayMode Guard Protection

Baked 기반 시스템에서 가장 흔하고 위험한 버그 중 하나가 **empty registry 문제**임.

개발자가 PlayMode 진입 전에 global registry를 갱신하지 않았을 때 발생함.

UNInject는 **Editor 전용 validation 시스템**을 통해 이를 방지함.

Play 버튼이 눌리면 다음 동작이 수행됨.

* `MasterInstallerPlayModeGuard`가 state change를 감지함
* global registry `_globalReferrals` 배열 크기를 검사함
* 비어 있을 경우 즉시 warning 로그를 출력함

이를 통해 개발자가 registry 갱신을 잊었을 경우, 바로 이를 인지할 수 있음. 

---

### Cache-Friendly Injection Execution

UNInject는 내부적으로 **cached structural mapping 구조**를 사용함.

global 및 scene registry는
미리 baked된 리스트 구조를 기반으로 생성되며  
`Awake` 시점에 빠른 `Dictionary<Type, Component>` lookup 구조로 변환됨.

이러한 **data-oriented 구조**는
scene hierarchy를 직접 순회하는 방식보다 훨씬 높은 성능을 제공함.

# API Reference & Usage

UNInject API는 **두 가지 핵심 개념**을 중심으로 설계되었음.

* **Providers (Referrals)**
* **Consumers (Injects)**

대부분의 DI API는 거대한 외부 Binder 스크립트를 요구하지만
UNInject는 **attribute 기반 선언형 pipeline**을 제공함.

이를 통해 dependency resolution을
**표현력 높게**, **안전하게**, 그리고 **매우 빠르게** 사용할 수 있도록 설계했음.  

**한 마디로** 사용자는 Scene에 해당 SDK가 제공하는 3개의 핵심 컴포넌트를 배치한 후,  
코드레벨에서 `[Inject], [Referral], [SceneReferral]` 등의 어트리뷰트 만으로 추가 Resolve 없이 DI 디자인이 가능한 셈. 

---

## The Provider Attributes

서비스는 referral attribute를 통해 프레임워크에 등록됨.

```csharp
[Referral(typeof(IInputService))]
public class DesktopInputManager : MonoBehaviour, IInputService { }
```

provider attribute는 다음을 보장함.

* **자동 Editor Scanning**
* **Interface Binding 지원**
* **중복 등록 방지**

지원되는 provider marker는 다음과 같음.

* `[Referral]` → Global / Master scope
* `[SceneReferral]` → Scene-local scope

---

## The Consumer Attributes

dependency는 consumer marker를 통해 target 클래스에 주입됨.

```csharp
public class PlayerController : MonoBehaviour
{
    // Resolved at editor-time (Baked)
    [Inject] [SerializeField] private Animator _animator;

    // Resolved at runtime from MasterInstaller
    [GlobalInject] private IInputService _input;

    // Resolved at runtime from SceneInstaller
    [SceneInject(optional: true)] private LevelManager _level;
}
```

이 설계를 통해 **field 선언만 읽어도**

* dependency의 **출처**
* dependency의 **lifecycle**

을 즉시 파악할 수 있음.

---

## Optional Dependencies (MOAK Obj Rogic)

Global 및 Scene injection은 `Optional` 속성을 지원함.

```csharp
[GlobalInject(optional: true)] 
private AnalyticsManager _analytics;
```

manager가 존재하지 않는 경우
(예: 독립 테스트 환경)

UNInject는 오류를 발생시키지 않고 injection을 건너뜀.

이는 **대규모 프로젝트 환경**이나 **모듈 테스트** 환경에서 매우 유용함.  

**Major DI 솔루션 (VContainer, Zenject 등)**들이 제공하던 MOAK 객체에 대한 의존성 요구와,  
디버깅 기능을 이 `[SceneInject](optional: true), [GlobalInject])(optional: true)` 를 통해 강력하게 제공함  

---

## Injected State Alarm

각종 `Function` 이라 불리우는 **Start**, **Awake** 와 마찬가지로, **UNInject** 또한 명시적인 주입 완료 시점 `Function`을 제공함.  

```csharp
public class PlayerController : MonoBehaviour, IInjected
{
    [GlobalInject] private IInputManager _input;                 
    [SceneInject(true)] private IStageContext _stageContext;    

    public void OnInjected()
    {
        // ObjectInstaller가 이 컴포넌트에 대해 주입을 수행했고,
        // "필수 의존성" 주입이 모두 성공했을 때 자동으로 호출됩니다.

        // 호출 주체: ObjectInstaller.TryInjectTarget() 내부에서 success == true이고  
        // 대상이 IInjected를 구현한 경우 OnInjected()를 호출합니다.
        
        // 호출 조건: 필수(Required) 의존성 주입이 모두 성공했을 때만 호출됩니다.  
        // (Optional로 표시된 의존성 누락은 성공/실패에 영향 없음)
        
        // 호출 시점/횟수: InjectTarget() / InjectGameObject() / Awake()에서  
        // 주입이 실행될 때마다(즉, 주입을 여러 번 호출하면) 그때마다 호출될 수 있습니다.
    }
}
```

(현재는 `IInjected` 인터페이스를 상속받아야만 해당 `OnInjected` 멤버를 사용 가능하지만,  
추후 Native 영역으로 이를 추출하여, 별도의 상속 없이도 `OnInjected` 멤버를 사용할 수 있도록 구조를 변경할 예정임.)

# Lifecycle & Internal Architecture

UNInject는 **엄격하게 정렬된 실행 파이프라인**을 기반으로
object lifecycle을 관리함.

```
Step 1: Master Initialization [Order: -1000]
------------------------------------------
MasterInstaller.Awake()
→ baked 리스트로부터 Global Registry 재구성

Step 2: Scene Initialization [Order: -900]
------------------------------------------
SceneInstaller.Awake()
→ baked 리스트로부터 Scene Registry 재구성

Step 3: Object Injection [Order: -500]
--------------------------------------
ObjectInstaller.Awake()
→ 모든 [GlobalInject], [SceneInject] 대상 수집
→ 사전 컴파일된 lambda setter 실행
```

---

# Dynamic Object Support

baked DI 시스템의 주요 한계 중 하나는
**동적으로 생성되는 객체 처리 문제**임.

UNInject는 이를 **명시적 injection API**를 통해 해결했음.

예를 들어 **PowerPool integration**이나
**factory 기반 runtime spawn 상황**에서는
다음 방식으로 의존성을 주입할 수 있음.

```csharp
// 1. Spawn from PowerPool
var handle = PowerPool.Spawn(enemyPrefab).Rent();

// 2. Inject runtime dependencies instantly
ObjectInstaller.Instance.InjectTarget(handle.Instance);
```

`InjectTarget` 메서드는 동일한 `TypeDataCache` 구조를 사용하기 때문에,  
대량의 동적 생성 객체에서도 **allocation 없이 매우 빠른 주입**이 가능함.

# Performance Comparison

**Zero Runtime GetComponent Lookup Cost**

UNInject는 `GetComponent` 체인을 근본적으로 대체하도록 설계되었음.

로컬 dependency는 전부 **Editor 단계에서 resolve**됨.

**"Bake Dependencies"** 과정에서
hierarchy를 스캔하고 reference를 scene 파일에 직접 serialize함.

runtime에서는 Unity의 native deserialization이 자동으로 연결을 복원.
**탐색을 위한 CPU 연산이 전혀 발생하지 않음**.

---

**Expression Tree Compilation vs Reflection**

일반적인 injection 시스템은
`FieldInfo.SetValue(target, value)` 방식을 사용함.

이 방식은 대량의 boxing allocation과 CPU 오버헤드를 발생시킴.

UNInject의 **TypeDataCache**는
타입당 한 번만 expression tree를 컴파일함.

```csharp
var assign = Expression.Assign(Expression.Field(typedTarget, field), typedValue);
var lambda = Expression.Lambda<Action<object, object>>(assign, targetParam, valueParam);
```

<img width="800" height="600" alt="Image" src="https://github.com/user-attachments/assets/964a75dc-0a7d-4c7c-9455-bec0fb48e0bd" />  

컴파일된 lambda는 **native C# 코드와 동일한 속도로 실행됨**.

한 번 캐싱되면
10,000개의 객체 injection도 **마이크로초 단위로 수행 가능함**.

이는 복잡한 UI 생성이나
대규모 레벨 인스턴스 생성 시 발생하는
**GC spike 및 frame drop 문제를 방지함**.

# Editor Case

> **bake되지 않은 master prefab 상태에서 PlayMode 진입 시** 다음 warning이 발생할 수 있음.

`[MasterInstaller] Global registry is empty. Did you forget to click 'Refresh Global Registry' before Play?`

: **LogWarning**

이는 `MasterInstallerPlayModeGuard`가 제공하는 **안전 장치로 정상 동작임**.

이 warning이 발생했다면 registry bake가 필요함.

시스템은 crash 방지를 위해 **null injection fallback**을 수행하지만
dependency는 정상적으로 충족되지 않음.

# 참고 및 향후 업데이트 방향성

### 현재 지원하는 컴파일러 

현재 **UNInject** 는 **IL2CPP**, **[A.O.T]** 형태의 컴파일을 지원하지 않고 있음.  
(물론 현재 미지원 컴파일에 관련해서 **명확한 풀백 로직**이 존재하며, 이 **풀백 로직** 또한 강력함.)

이것의 이유는 단순하게, `동적 코드 생성` 의 대체제를 개발하는 과정에 있기 때문임.  
물론 지원 컴파일 외 환경에서 `의존성 매칭 비용` 과 `동적 코드 생성`의 비용이 거의 유사하기 때문에  
엔터프라이즈 도약을 위해서는 결국, `동적 코드 생성 및 매칭` 시스템이 필수 불가결함.  

따라서 향우 업데이트 방향성은 다음과 같음.

```
1.1 (성능 “약속”을 IL2CPP까지 확장하는 릴리즈)
런타임 리플렉션/Expression 의존 최소화: “에디터 베이크 기반 Inject Plan”
방향 1) 코드 생성(Source Generator/Editor tool): 타입별 Inject(target) 메서드 생성
방향 2) 베이크된 테이블(ScriptableObject): 필드 세터를 ID 기반으로 맵핑(IL2CPP 친화)
목표: **모든 런타임에서 “SerializeField 급 비용”**에 근접하도록 보장
```

```
1.2 (아키텍처 고도화: Scope/Lifetime)
Scope 개념 도입(VContainer LifetimeScope 를 “유니티 친화 모델”로 단순화)
Global(DDOL), Scene(씬), Object(루트)
런타임 Register API는 여기서 “소유권/언레지스터/씬 전환 정책”과 함께 제공
```

```
2.0 (엔터프라이즈 완성형)
검증/진단 체계(그래프/리포트/빌드 전 검사)
프로파일링 툴링(주입 비용/누락/중복 시각화)
Factory 패턴 또는 “유니티 오브젝트 생성 파이프라인”과의 통합
```

### 마치며

여러 **DI Solution SDK**를 경험하면서 고질적으로 발생한 문제를 수집하였고,  
이를 개선하기 위해 여러 주변 개발자분들의 케이스를 반영하여 `라이트하고 쉬우며, 성능이 극대화된` DI 솔루션을 만들게 되었음.  

기존 DI 솔루션들의 문제점은 **학습곡선**, **Life Cycle 관리 모호**, **초기 아키텍쳐 설계 비용 높음** 등임.  

기존 Major 솔루션들의 장점은 보존하고, 학습곡선과 초기 설계 비용을 크게 낮춘 좋은 구조이며,  
실제로 필자는 이 SDK를 캐주얼, 미드코어 개발에 필수적으로 활용함.  

기회가 되면 꼭 써보길 권고드림.  

특히 `IInjected` 인터페이스의 `OnInjected` 멤버 펑션은 매우 강력함. 객체 라이프사이클 개념에서 매우 유용한 멤버임.
